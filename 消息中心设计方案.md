# 消息中心设计方案

## 1. 目标

在现有系统中建立一套统一的消息中心，先落地系统内消息，后续可平滑扩展到钉钉、飞书、邮件、短信等外部通道。

当前系统已经具备部分基础能力：

- `sys_notice` 表与 `NoticesService`
- workflow 审批任务通知
- workflow 抄送节点通知
- workflow 通知节点通知

但还缺少完整的产品层能力：

- 我的消息中心
- 未读/已读管理
- 消息分类与来源标识
- 跳转到业务对象页
- 顶栏铃铛图标与未读数
- 流程结果通知
- 统一的消息投递服务

本方案目标是把这些能力收成一个统一体系。

---

## 2. 设计原则

### 2.1 先做站内消息，后扩通道

当前阶段先只做系统内消息，不把复杂度扩到钉钉/飞书实时推送和模板渠道上。

但数据模型和服务层必须预留：

- 渠道字段 `channel`
- 外部发送状态 `externalStatus`
- 外部消息标识 `externalTraceId`

### 2.2 公告和消息分开

建议保留现有 `sys_notice` 作为系统公告或历史兼容能力。

新增独立的消息中心表：

- 公告：面向全局/大范围广播
- 消息：面向单用户、单业务事件、单流程事件

### 2.3 workflow 是消息的重要来源，但不是唯一来源

消息中心不只服务 workflow，后续也应支持：

- 业务对象提醒
- 系统提醒
- 外部回调通知
- AI/机器人通知

所以消息中心必须是独立能力，而不是挂在 workflow 模块里的附属表。

### 2.4 先做轮询，不急着上 Socket.IO

当前阶段不把 Socket.IO 作为第一优先级。

先做：

- 顶栏铃铛
- 未读数量接口
- 我的消息中心页面
- 页面内轮询未读数

等站内消息闭环稳定后，再决定是否上实时推送。

---

## 3. 推荐方案总览

### 推荐方案

建立统一消息中心，包含：

1. 独立消息表 `sys_message`
2. 统一服务 `MessageCenterService`
3. workflow 消息统一接入消息中心
4. 顶栏铃铛入口
5. 我的消息中心页面
6. 未读/已读能力
7. 跳转到业务页/实例页

### 当前阶段不做

1. Socket.IO 实时推送
2. 钉钉/飞书真正发送
3. 邮件/短信真正发送
4. 复杂的模板管理后台

这些先在设计上预留，不作为第一阶段上线内容。

---

## 4. 数据模型设计

### 4.1 新增消息表 `sys_message`

推荐新增独立表，而不是继续复用 `sys_notice`。

#### 建议字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | varchar | 主键 |
| title | varchar | 消息标题 |
| content | text | 消息内容 |
| messageType | varchar | 消息类型 |
| sourceType | varchar | 来源对象类型 |
| sourceId | varchar | 来源对象ID |
| receiverId | varchar | 接收人ID |
| senderId | varchar | 发送人ID，可空 |
| isRead | char(1) | 是否已读 |
| readTime | datetime | 已读时间 |
| channel | varchar | 通知通道 |
| linkType | varchar | 跳转类型 |
| linkUrl | varchar | 跳转地址 |
| linkParams | json | 跳转参数 |
| extraData | json | 扩展数据 |
| isActive | char(1) | 是否有效 |
| createTime | datetime | 创建时间 |
| updateTime | datetime | 更新时间 |
| createUser | varchar | 创建人 |
| updateUser | varchar | 更新人 |
| isDelete | datetime/null | 软删时间 |

#### 建议 messageType 枚举

- `workflow_task`：审批任务通知
- `workflow_cc`：流程抄送通知
- `workflow_notification`：流程通知节点消息
- `workflow_result`：流程结果通知
- `system_notice`：系统公告转个人提醒（可选）
- `business_alert`：业务提醒

#### 建议 sourceType 枚举

- `workflow_instance`
- `workflow_task`
- `project`
- `task`
- `ticket`
- `change`
- `customer`
- `system`

#### 建议 channel 枚举

当前阶段只用：

- `system`

后续预留：

- `dingtalk`
- `feishu`
- `email`
- `sms`

#### 建议 linkType 枚举

- `route`
- `url`

## 5. 后端服务设计

### 5.1 新增 `MessageCenterService`

建议新增统一服务：

- `nest-admin/src/modules/messages/service.ts`

#### 核心职责

1. 创建系统内消息
2. 查询消息列表
3. 查询未读数
4. 标记已读/全部已读
5. 封装 workflow 消息创建逻辑
6. 为未来外部通道预留分发接口

#### 建议接口

```ts
sendSystemMessage(dto)
sendWorkflowTaskMessage(dto)
sendWorkflowCcMessage(dto)
sendWorkflowNotificationMessage(dto)
sendWorkflowResultMessage(dto)

getUserMessages(userId, params)
getUnreadCount(userId)
markAsRead(messageId, userId)
markAllAsRead(userId)
```

### 5.2 workflow 集成点

当前 workflow 消息主要来自：

1. 审批任务创建时
2. 抄送节点执行时
3. 通知节点执行时
4. 流程完成/拒绝时（建议新增）

#### 审批任务创建通知

当前是在：
- `workflow/service.ts` 中直接调用 `NoticesService.add()`

建议改成：
- 调 `MessageCenterService.sendWorkflowTaskMessage()`

消息内容建议包括：

- 流程名称
- 节点名称
- 业务对象标题
- 业务对象编号
- 发起人
- 跳转业务页链接

#### 抄送节点通知

当前在：
- `cc-node.handler.ts`

建议改成：
- 调 `MessageCenterService.sendWorkflowCcMessage()`

#### 通知节点通知

当前在：
- `notification-node.handler.ts`

建议改成：
- 调 `MessageCenterService.sendWorkflowNotificationMessage()`

#### 流程结果通知

这是当前缺口，建议新增：

在 `workflow-integration.service.ts` 或 `workflow/service.ts` 完成/拒绝后增加：

- 发起人收到审批通过通知
- 发起人收到审批拒绝通知
- 相关抄送人收到结果通知（可选）

## 6. API 设计

### 6.1 新增消息接口

建议新增模块：

- `GET /system/messages/list`
- `GET /system/messages/unread-count`
- `POST /system/messages/read/:id`
- `POST /system/messages/read-all`

#### 列表返回建议

每条消息返回：

```json
{
  "id": "...",
  "title": "项目立项审批待处理",
  "content": "项目 PRJ-20260414-0002 已提交审批，请及时处理。",
  "messageType": "workflow_task",
  "sourceType": "project",
  "sourceId": "6",
  "receiverId": "1",
  "isRead": "0",
  "channel": "system",
  "linkType": "route",
  "linkUrl": "/projectManage/detail",
  "linkParams": {
    "id": "6",
    "taskId": "12",
    "instanceId": "1001",
    "fromWorkflow": "1"
  },
  "createTime": "2026-04-15 10:00:00"
}
```

### 6.2 workflow 跳转参数约定

建议统一 workflow 消息跳转参数：

- `id`
- `taskId`
- `instanceId`
- `fromWorkflow=1`

这样前端业务页可以继续复用当前已经做好的统一审批面板。

## 7. 前端设计

### 7.1 顶栏铃铛入口

位置：

- `nest-admin-frontend/src/layout/components/TopBar/index.vue`

建议放在：

- **用户头像左侧**

表现建议：

1. 铃铛图标
2. 未读数 badge
3. 点击展开最近消息下拉
4. 底部“查看全部消息”

#### 最近消息下拉内容

建议展示最近 5~10 条：

- 标题
- 摘要
- 时间
- 已读/未读状态

点击后：
- 标记已读
- 跳转对应业务页或实例页

### 7.2 我的消息中心页

建议新增页面：

- `nest-admin-frontend/src/views/system/messageCenter/index.vue`

#### 页面结构建议

顶部筛选：

- 全部
- 未读
- 审批通知
- 抄送通知
- 流程结果通知
- 系统公告

消息列表字段：

- 类型图标
- 标题
- 摘要
- 时间
- 已读状态

操作：

- 标记已读
- 全部已读
- 跳转业务对象

### 7.3 workflow 与消息中心的关系

workflow 页面继续保留：

- 我的待办
- 我参与 / 我发起实例

消息中心负责：

- 查看通知类信息
- 提醒类入口

两者不是替代关系，而是互补关系。

## 8. 推荐交互流程

### 8.1 审批任务消息

1. 某流程到审批节点
2. 系统创建待办
3. 同时创建一条消息 `workflow_task`
4. 顶栏铃铛未读数 +1
5. 用户点消息
6. 跳到业务页审批

### 8.2 抄送消息

1. 流程到抄送节点
2. 系统创建抄送消息 `workflow_cc`
3. 用户在消息中心里查看
4. 点击跳到业务对象详情/流程实例详情

### 8.3 流程结果消息

1. 流程完成 / 被拒绝
2. 系统给发起人发 `workflow_result`
3. 消息中心显示结果
4. 点击可跳到业务对象页或实例详情页

## 9. Socket.IO 是否需要

### 当前阶段建议

**不作为第一优先级。**

先做：

1. 统一消息模型
2. 顶栏铃铛
3. 我的消息中心
4. 未读数接口
5. 前端轮询未读数

### 轮询建议

当前完全够用的方案：

- 登录后每 30~60 秒拉一次未读数
- 打开消息中心时拉完整列表
- 审批/转交/加签等动作完成后主动刷新未读数

### 什么时候再上 Socket.IO

当你满足以下条件时再上：

1. 消息中心已稳定
2. 待办/消息必须秒级到达
3. 多标签页/多人协同提醒的实时需求明确

### 后续如果扩到钉钉/飞书

推荐模式是：

- **站内消息始终是主记录**
- 钉钉/飞书只是附加通道

也就是：
1. 先存系统内消息
2. 再尝试外部发送
3. 外部失败不影响站内消息可用

这样最稳。

## 10. 实施顺序建议

### Phase 1：最小可行版本

1. 新建 `sys_message` 表
2. 新增 `MessageCenterService`
3. 把 workflow 的审批通知 / 抄送通知 / 通知节点统一接入消息中心
4. 增加流程结果通知
5. 顶栏加铃铛 + 未读数
6. 新增消息中心页
7. 前端消息点击跳业务页

### Phase 2：体验增强

1. 消息已读/全部已读
2. 消息分类
3. 最近消息下拉
4. workflow 消息模板优化

### Phase 3：外部通道扩展

1. 钉钉发送器
2. 飞书发送器
3. 邮件 / 短信发送器
4. 通道级失败重试和日志

---

## 11. 最终推荐结论

从系统整体角度，最推荐的方案是：

**建立统一消息中心，先只做站内消息；workflow 所有通知统一接入消息中心；在用户头像左侧放铃铛图标显示未读消息；后续通过通道层平滑扩展到钉钉/飞书。**

这套方案的优点：

1. 当前阶段复杂度可控
2. 和现有 workflow 主链路自然衔接
3. 用户入口清晰
4. 后续扩展不需要推翻重来
